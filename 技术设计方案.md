# v3-admin-vite 技术设计方案

## 1. 架构 (Architecture)

本项目采用现代化的前端技术栈构建，专注于提供高性能、高可维护性的后台管理系统解决方案。整体架构分层清晰，职责明确。

### 1.1 技术栈选型

- **核心框架:** Vue 3 (v3.5+) - 采用 Composition API 组织逻辑，提升逻辑复用与代码可读性。
- **构建工具:** Vite (v7+) - 提供极速的冷启动和热重载(HMR)体验。
- **开发语言:** TypeScript - 提供静态类型检查，增强代码健壮性和可维护性。
- **路由管理:** Vue Router (v4) - 负责前端页面的路由分发和权限拦截。
- **状态管理:** Pinia - 作为 Vue 的专属状态管理库，轻量且完备支持 TypeScript。
- **环境要求:** Node.js v24.12.0 - **必须**通过 `nvm use 24` 切换至此版本，确保依赖行为一致。

### 1.2 架构分层

系统主要分为以下几层：

1.  **视图层 (View Layer):** 负责页面的展示和交互，包含页面 (Pages) 和组件 (Components)。
2.  **状态层 (State Layer):** 基于 Pinia 管理全局共享状态（如用户信息、权限字典等）。
3.  **服务层 (Service Layer/API):** 统一放置在 `src/common/apis`，负责具体业务的请求参数处理及接口调用逻辑。
4.  **网络层 (HTTP Layer):** 统一放置在 `src/http`，封装底层 HTTP 请求（Axios），处理请求/响应拦截、全局异常及 Token 管理。

## 2. 系统组件 (System Components)

系统的组件体系明确划分了基础 UI 组件和业务复用组件，保障页面样式统一与开发效率。

### 2.1 核心组件库选型与使用边界

- **Element Plus:** 作为基础 UI 组件库，用于**除了表格之外**的所有 UI 交互和布局需求（如按钮、表单、弹窗、布局等）。禁止使用 Element Plus 的表格组件 (`el-table`)。
- **vxe-table:** 作为专业的表格解决方案，**必须且仅用于**所有表格相关功能的场景。禁止在非表格场景滥用 vxe 体系的非表格组件（如 `vxe-button`、`vxe-form` ）。

### 2.2 核心业务组件

系统内部封装了多个核心业务组件，以应对复杂的业务场景和整体性能瓶颈：

- **标准页面布局组件 (Standard Page / SearchTablePage):**
  位于 `src/pages/components/SearchTablePage.vue`。系统内标准的列表页**必须**采用此组件。该组件规定了标准页面的三大核心区块：
  1. **搜索表单区:** 支持筛选项的分类展示与折叠收起，标准表单项由 Element Plus 构建。
  2. **操作按钮区:** 支持灵活自定义页面级的各类操作按钮（如新增、导出、批量等）。
  3. **数据表格视图:** 内部嵌套并统一了 `vxe-table` 的布局及分页行为，不再允许在列表页直写 `vxe-table` 标签。

- **高性能下拉选择组件 (Virtual Select):**
  专为选项超过 100 条的大规模数据场景封装的下拉组件。
  - **多维搜索支持:** 必须同时支持按展示文本 (`label`) 和实际绑定值 (`value`) 检索。
  - **DOM 渲染优化:** 解决传统 Select（由于底层采用 `v-show` 全量保留 DOM）在数据庞大时引发的页面卡顿问题。要求内部必须采用虚拟渲染 (Virtual Scroll) 的方式封装，仅保持可视范围的 DOM 节点。

- **自定义级联选择组件 (Custom Cascader):**
  针对级联选择场景进行特定封装。自身闭环处理特定的级联逻辑（如数据层级深、特殊异步加载等），确保级联交互体验的最佳化和数据流动的可控性。

- **树形数据展示组件 (Tree Component):**
  专门针对具有显著层级结构（例如：系统菜单管理、组织部门管理、权限配置树等场景）进行系统级抽象与封装的组件体系。
  - **标准化交互:** 统一封装包含一键全量展开/折叠、本地搜索过滤、以及多选状态下父子节点级联响应在内的核心逻辑。
  - **灵活性与渲染:** 与外部系统解耦，自包含将扁平列表转化为树形数据结构的解析逻辑；同时兼具大数据下的性能承载能力。

## 3. Util 工具类 (Utilities)

为了提高代码复用率、降低各个模块之间的强耦合，并统一核心底层逻辑，系统应规范化抽离和管理所有的工具类函数 (Utils)。工具函数通常应保持纯函数特性（即不产生副作用、输出仅受输入决定）。

### 3.1 分类与职责

- **通用基础工具 (Common Utils):** 沉淀与业务弱相关的底层方法。
  - **统一非空判定:** **必须**提供并使用统一的非空校验工具函数，需能够准确识别并判定以下场景为“空”：`null`、`undefined`、空对象 `{}`、空数组 `[]` 以及空字符串 `''`。
  - **常规操作:** 包含字符串操作、高精度类型判断、时间日期处理（统一基于 `dayjs`）、深拷贝、防抖与节流等。
- **业务辅助工具 (Business Utils):** 存在业务特性的公共逻辑封装，例如特定状态字典的映射与转换、数据格式归一化清洗、或者是部分复杂的表单默认校验规则集合。

### 3.2 管理与导出规范

- **存放位置:** 所有的项目级工具函数强制统一存放在类似 `src/utils` 等公共的专用目录下，禁止在页面或组件级别目录下随意封装全系统范围流通的通用工具。
- **具名导出 (Named Export):** 为获得 Vite 构建过程中的最佳 Tree-Shaking（摇树优化）支持，避免打包体积冗杂，工具文件内的重要函数均应采用具名导出 (`export const fnName = () => {}`)，禁用单一巨大的 default 对象包裹。
- **内联注释 (JSDoc):** 作为高频被复用的工具函数，必须硬性要求携带完整的中文 JSDoc 块级注释，精确阐明其功能目的、必要参数及返回判定。

## 4. 书写规范 (Coding Conventions)

为了保证代码的一致性并降低维护成本，在开发过程中需严格遵守以下强制性规范：

### 3.1 页面与脚本规范

- **Setup 语法:** 创建 Vue 页面或组件时，**必须统一使用** `<script setup></script>`，禁止添加 `lang="ts"` 属性（即不允许使用 `<script setup lang="ts">`）。

### 3.2 接口与网络请求规范

- **统一发送:** 所有 HTTP 请求**禁止**直接使用 `fetch` 或未经封装的 `axios`，**禁止**在页面或组件内直接实现请求细节。必须通过导入 `src/http` 暴露的实例进行操作。
- **集中管理:** 所有业务接口定义**强制**放置在 `src/common/apis` 目录下。
- **单模块单文件:** 每个 API 文件对应一个独立页面或业务模块（例如 `user.ts` 对应用户模块），文件名需语义化。
- **命名与导出:** 接口方法名称须通过动宾结构准确表达业务含义（如 `getUserInfo`），并进行明确的命名导出 (named export)。

### 3.3 注释与文档规范

- **中文注释:** 在编写业务逻辑注释及生成的配套文档时，**必须**使用中文，确保语义明了。禁止编写无意义或冗余的英文注释（业内公认的专有技术词汇除外）。
